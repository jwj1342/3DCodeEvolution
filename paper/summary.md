## Overview & Mapping to the TAC-Graph Axes

The **Temporal-Agentic-Contextual Code Graph (TAC-Graph)** is a unified representation that fuses three orthogonal views of a codebase into a single tensor `T ∈ R^(H × A × S)`. The **X / Temporal axis** encodes the macro-evolution of the project as mined from git history (commits, diffs, bug-fixes, features, branches); the **Y / Agentic axis** encodes *how* the code came to be, as the trajectory of a coding agent (chain-of-thought, read/write/search operations, bash runs, and debug trial-and-error); and the **Z / Spatial axis** encodes the current semantic state of the code (AST, data- and control-flow graphs, file dependencies, and API/call graphs). Two methodological commitments tie the axes together: a *cross-dimension projection* that injects git and agent history into a "history-aware AST," and *temporal graph learning* (Temporal GNNs / Graph Transformers) that reconciles the mismatched time granularities of agent operations (seconds) and commits (days). On top of this representation, TAC-Graph proposes two delivery **forms**: **Form A**, an explicit GraphQL-like DSL through which an agent issues `Query(Target, History, Agent_Trace)` to pull precise context, and **Form B**, an implicit Graph-VAE that compresses the whole tensor into a latent embedding loaded as soft-prompts / prefix tuning to bypass the context window. The downstream payoffs are *Task Untangling* (splitting tangled bug-fix+feature commits) and *Trajectory Optimization* (pruning agent search by contrasting successful vs. failed 3D subgraphs).

This survey maps the literature onto these axes and forms. The remainder of this section gives the reader a high-level map of which research communities feed which part of the design before later sections drill into each.

### The three axes and their source communities

**Z / Spatial axis — code structure (SE structure-aware modeling + GNN-for-code).** The strongest body of prior work concerns representing code as structure. Foundational graph-and-tree encoders ground this axis: *Learning to Represent Programs with Graphs* [1711.00740] fuses AST with data/control-flow edges and runs GGNNs over the result (Theory/inspiration), GraphCodeBERT [2009.08366] bakes data-flow into pre-training (borrowed Method / Baseline), and code2vec [1803.09473], UniXcoder [2203.03850], StructCoder [2206.05239], AST-T5 [2401.03003], TreeBERT [2105.12485], CodeBERT [2002.08155] and CodeT5 [2109.00859] supply complementary AST/path/identifier-aware encoders (mostly Baseline; AST-T5/StructCoder are borrowed Method for the history-aware AST). PERFOGRAPH [2306.00210] and Devign [1909.03496] show how to enrich program graphs with numerical/semantic node-edge types and jointly encode AST+CFG+DFG with GNNs (borrowed Method; Devign also a Baseline/Benchmark). At repository scale, a large retrieval-augmented sub-community instantiates the Z-axis for code completion: RepoCoder [2303.12570], CoCoMIC [2212.10007], RepoFusion [2306.10998], repository-level prompt generation [2206.12839], A³-CodGen [2312.05772], Repoformer [2403.10059], R2C2-Coder [2406.01359], and graph-based variants RepoHyper [2403.06095], GraphCoder [2406.07003] and DraCo [2405.19782] (all borrowed Method; several also Benchmark). RepoBench [2306.03091] and CrossCodeEval [2310.11248] are the canonical Benchmarks here. API/knowledge-graph grounding — DocPrompting [2207.05987], ToolCoder [2305.04032], programming knowledge graphs [2410.18251], and the HarmonyOS KG synthesis work [2512.00380] — feeds the call/API-graph layer (borrowed Method); CodeRAG-Bench [2406.14497] is the retrieval Benchmark.

**Y / Agentic axis — coding-agent trajectories (LLM-agent community).** This axis draws on coding-agent systems and agent-memory research. SWE-bench [2310.06770] (Benchmark) defines the issue→locate→patch task, and SWE-agent [2405.15793], AutoCodeRover [2404.05427], Agentless [2407.01489], OpenHands [2407.16741], CodeAct [2402.01030], CodeAgent [2401.07339], LingmaAgent [2406.01422], RepoGraph [2410.14684], SWE-Gym [2412.21139] and Lingma SWE-GPT [2411.00622] are concrete sources of read/write/search/test traces (Baseline; SWE-Gym/CodeAct also supply trajectory training data). The *structure* and *replay* of those trajectories come from the agent-memory line: Reflexion [2303.11366], Voyager [2305.16291], ExpeL [2308.10144], Agent Workflow Memory [2409.07429], Synapse [2306.07863], LATS [2310.04406], FireAct [2310.05915], Generative Agents [2304.03442] and AriGraph [2407.04363] (Theory/inspiration; AWM/Synapse/AriGraph are borrowed Method for indexing traces), with AgentBench [2308.03688] as an evaluation Benchmark. HCI eye-tracking and IDE-telemetry studies — follow-up attention [2210.05506], EyeTrans [2402.14096], developer-behavior studies [2405.16081, 2403.08480, 1903.03358], Code Compass [2405.06271], and the large-scale AI-IDE session analysis [2604.00436] — empirically ground what an agent/developer trajectory actually looks like (Theory/inspiration; [2403.08480] contributes a behavioral Metric).

**X / Temporal axis — git history (MSR + dynamic-graph learning).** Two communities feed this axis. The mining-software-repositories side supplies code-change representations: CC2Vec [2003.05620], CCRep [2302.03924], CCT5 [2305.10785], CCBERT [2309.15474], CommitBERT [2105.14242], CodeReviewer [2203.09095], *Learning to Represent Edits* [1810.13337], MODIT [2108.06645], CoditT5 [2208.05446], NMT-for-code-changes [1901.09102], and general-purpose change embeddings [2106.02087] (borrowed Method / Baseline), plus defect-prediction signals [2403.11079, 2410.12107, 2110.05371], SZZ evaluation [2308.05060] (Method/Baseline + a recall Metric), and the DeepSoft vision [1608.00092] and the MSR-with-LMs survey [2604.00787] (Theory/inspiration). Evolving-API knowledge graphs KCoEvo [2603.07581] and KG-based repo-level generation [2505.14394] tie API version transitions to the temporal axis (borrowed Method). The *machinery* for learning over this evolving graph comes from the dynamic-graph community: TGAT [2002.07962] and TGN [2006.10637] are the anchor methods, with DyRep [1803.04051], JODIE [1908.01207], EvolveGCN [1902.10191], GraphMixer [2302.11636], ROLAND [2208.07239], DyGFormer [2303.13047], and the granularity study [2311.12255] addressing the seconds-vs-days mismatch directly (borrowed Method / Baseline; DyGFormer ships DyGLib as a Benchmark). Two surveys [1905.11485, 2304.05729] provide the taxonomy (Theory/inspiration). The visualization community — M3triCity [2204.10006], the modern-software-visualization SLR [2003.00643], and code-city tools [2408.08141, 2408.16777, 2412.16061, 2504.18238] — offers prior art for jointly rendering structure and evolution in 3D, with comprehension theory [2310.11301, 2304.09301] motivating what such views must surface (Theory/inspiration; SecCityVR/M3triCity are inspiration for the spatial-temporal overlay).

### The two delivery forms

**Form A — GraphQL-like query DSL.** The retrieval-augmented and agent-tooling work above doubles as prior art for Form A: selective and graph-based retrieval (Repoformer [2403.10059], RepoHyper [2403.06095], GraphCoder [2406.07003], DraCo [2405.19782]), tool-mediated repo navigation (RepoGraph [2410.14684], CodeAgent [2401.07339], ToolCoder [2305.04032]), and trajectory-as-exemplar memory retrieval (Synapse [2306.07863], AriGraph [2407.04363]) all model "query precise cross-axis context on demand" (borrowed Method). CodeRAG-Bench [2406.14497] and CrossCodeEval [2310.11248] are the natural Benchmarks for measuring query quality.

**Form B — Graph-VAE soft-prompt.** The implicit form rests on two pillars. The *graph-compression* pillar is anchored by Variational Graph Auto-Encoders [1611.07308] and GraphVAE [1802.03480] (borrowed Method / Baseline), with DiffPool [1806.08804], graph transformers [2012.09699, 2106.05234, 2205.10803 GraphMAE, 2304.04779 GraphMAE2, 2205.12454 GraphGPS, 2007.02835 GROVER] and the graph-SSL survey [2103.00111] supplying encoder and self-supervision recipes. The *injection* pillar is the soft-prompt / context-compression line: Prefix-Tuning [2101.00190], Prompt Tuning [2104.08691], and P-Tuning v2 [2110.07602] are the anchor mechanisms (borrowed Method / Baseline), and Gist tokens [2304.08467], AutoCompressors [2305.14788], ICAE [2307.06945], LLMLingua [2310.05736], xRAG [2405.13792], Knowledge Prompts [2210.04726] and SoftPromptComp [2404.04997] show how large structured contexts can be distilled into a handful of latent tokens (borrowed Method).

### Downstream tasks

*Task Untangling* is served directly by the commit-untangling community: EpiceaUntangler [1502.06757], the fine-grained tangling dataset [2011.06244], LLM-based detection [2505.08263], multi-label concern detection [2601.21298], and the multi-agent Atomizer [2601.21298 / 2601.01233] (Benchmark + borrowed Method). *Trajectory Optimization* draws on search-based agents (LATS [2310.04406], LingmaAgent [2406.01422]) and contrastive success/failure memory (Reflexion [2303.11366], ExpeL [2308.10144], AWM [2409.07429]) (Theory/inspiration).

### The gap this RP fills

Each axis is mature in isolation — MSR has rich diff/commit encoders, the agent community has trajectory data and memory, and SE/GNN research has deep structural encoders — and pairs of axes occasionally meet (e.g., MODIT [2108.06645] fuses commit-intent with code structure; LingmaAgent [2406.01422] and RepoGraph [2410.14684] fuse agent search with structure; code-city tools [2408.08141] overlay evolution on structure). **No prior work, however, fuses all three axes — git evolution, agent trajectory, and code structure — into a single representation for coding agents, nor reconciles their mismatched time granularities within one temporal-graph model.** TAC-Graph targets exactly this gap, contributing the unified tensor, the two delivery forms, and the cross-dimension projection that the surveyed-but-siloed communities have not yet attempted jointly.

## AI / NLP Perspective: Repository-Aware & Agentic Code Generation

How a coding agent perceives a repository today is, almost universally, through *retrieval into a flat context window*. The papers reviewed here progressively enrich what gets retrieved — from in-file text, to cross-file snippets, to typed code graphs, to agent action traces — yet they share a common limitation: the repository is reconstructed anew on every query, and the temporal (git) and agentic (trajectory) signals that produced the code are discarded. We group the AI/NLP corpus into four themes and, for each, flag the value of each paper or cluster to TAC-Graph as a borrowed Method, Benchmark, new Metric, Baseline, or Theory/inspiration.

### Repository-level retrieval and the Spatial axis: from text snippets to typed code graphs

The dominant paradigm for repository perception is retrieval-augmented generation. The first wave treats the repo as a corpus of text to be retrieved by similarity. RepoCoder [2303.12570] couples a similarity retriever with a code LM in an iterative retrieve-then-generate loop and ships RepoEval; CoCoMIC [2212.10007] retrieves cross-file context via its CCFINDER tool and jointly models in-file and cross-file signals; and the repository-level prompt-generation framework of [2206.12839] learns a policy over which snippets to include for black-box LLMs. RepoFusion [2306.10998] (with the Stack-Repo dataset) and A^3-CodGen [2312.05772] organize retrieval by context *type* — imports, class hierarchies, local/global/library tiers — foreshadowing the layered structure of TAC-Graph's Z axis. These are **Baselines** for "how much does flat cross-file retrieval help," and their datasets (RepoEval, Stack-Repo) are reusable **Benchmarks**; the tiered-context decomposition is borrowable as a **Method** for partitioning the spatial layer.

The second wave makes retrieval *structure-aware* by building an explicit code graph and retrieving over it, which is the closest prior art to TAC-Graph's spatial slice. RepoHyper [2403.06095] constructs a Repo-level Semantic Graph and applies Search-Expand-Refine with link prediction; GraphCoder [2406.07003] builds a Code Context Graph fusing control-flow, data-dependence, and control-dependence edges with coarse-to-fine retrieval; DraCo [2405.19782] derives a repo-specific graph via extended dataflow analysis (and contributes the ReccEval benchmark); and Repoformer [2403.10059] adds *selective* retrieval, letting the model self-evaluate when retrieval will help (relevant to our Trajectory Optimization goal of pruning wasteful lookups). These supply borrowed **Methods** for graph construction and graph-expansion retrieval, and serve as strong **Baselines** that TAC-Graph extends by adding temporal and agentic axes on top of the same graph. R2C2-Coder [2406.01359] is both a context-assembly **Method** and a perturbation-based **Benchmark** (R2C2-Bench).

Crucially, this theme also gives us our evaluation scaffold. RepoBench [2306.03091] and CrossCodeEval [2310.11248] are the canonical cross-file completion **Benchmarks** with three-part task definitions (retrieve, predict next line, combined) and static-analysis-based identification of where cross-file context is required — directly applicable to evaluating TAC-Graph's Form A (DSL querying). CrossCodeEval additionally contributes a retriever-quality **Metric** showing top models still fail when dependencies are involved. The shared takeaway for TAC-Graph: today's agents perceive structure only as *retrieved snippets reconstructed per query*; TAC-Graph proposes a persistent, queryable graph that also carries the history of how each node came to be.

### Coding agents and the Agentic axis: how agents act on a repository

Coding agents formalize repository perception as a sequence of read/search/edit/test actions. SWE-bench [2310.06770] defines the issue→localize→patch **Task** over 2,294 real GitHub issues and is the field's reference **Benchmark**; it directly motivates our downstream Task Untangling and Trajectory Optimization framing. SWE-agent [2405.15793] shows that the agent-computer interface itself shapes behavior, and its structured interaction traces are a concrete **Method/Baseline** and a data source for the Y axis. Agentless [2407.01489] provides an interpretable localize-repair-validate **Baseline** whose clean trace is ideal seed data for the agentic axis, while AutoCodeRover [2404.05427] navigates via AST class/method structure and spectrum-based fault localization, bridging the Y and Z axes. OpenHands [2407.16741], CodeAgent [2401.07339], and Alibaba LingmaAgent [2406.01422] (which builds a repo knowledge graph and explores it via Monte Carlo tree search) are **Methods/Baselines** and, importantly, *sources of recordable agent trajectory data* — read/write/search/bash events — that the Y axis is designed to index. RepoGraph [2410.14684] is a plug-in **Method** that grounds agent navigation in a repo-level code graph, paralleling Form A's goal of letting agents pull precise structural context.

A second cluster turns trajectories into training signal. CodeAct [2402.01030] unifies the action space as executable code (with the CodeActInstruct dataset), SWE-Gym [2412.21139] releases real agent trajectories with executable runtimes and verifiers, and Lingma SWE-GPT [2411.00622] trains on development-process-centric data (thought, tool use, commit patterns) — collectively a **Method/Benchmark/Baseline** bundle and the empirical evidence that *encoding agent trajectories improves task resolution*, which is the central hypothesis of our Y axis. AgentBench [2308.03688] adds a general-purpose agent-evaluation **Benchmark** and a failure-mode taxonomy (long-term reasoning, instruction-following) useful as a **Metric** lens for trajectory quality. The state of the art: agents perceive a repo through trial-and-error action loops that are *executed and then thrown away*; TAC-Graph proposes to retain those loops as a first-class, contrastable axis.

### Agent trajectory, memory, and experiential learning

This theme supplies the abstractions for *storing and reusing* the agentic axis. Reflexion [2303.11366] stores verbal self-reflections to guide future attempts and explicitly contrasts successful vs. failed attempts — a near-direct **Method/inspiration** for our Trajectory Optimization (pruning search by contrasting successful vs. failed 3D subgraphs). Voyager [2305.16291] (skill library of reusable code), ExpeL [2308.10144] (extracting reusable NL insights from experience), Agent Workflow Memory [2409.07429] (inducing reusable workflows), and Synapse [2306.07863] (trajectory-as-exemplar with memory-indexed retrieval) are borrowed **Methods** for compressing, indexing, and replaying trajectories — Synapse's state-abstraction-then-retrieve scheme directly informs Form A's querying of agent traces. LATS [2310.04406] gives a tree-structured **Method** (MCTS + self-reflection) for representing branching agent reasoning, matching the trial-and-error structure the Y axis must encode. Generative Agents [2304.03442] contributes the memory-reflection-planning **Method** for organizing experience, and AriGraph [2407.04363] fuses episodic and semantic memory into a *dynamically updated knowledge graph world model* — the structural template closest to a queryable, graph-shaped agent memory (Form A) and to the episodic/semantic distinction our success-vs-failure contrast relies on. FireAct [2310.05915] is a **Method/Benchmark** showing trajectories can be harvested to fine-tune agents, informing how Y-axis data feeds Form B. These works are largely **Theory/inspiration plus borrowable Methods**: they prove agent experience is reusable but treat it as unstructured text or task-local memory, whereas TAC-Graph proposes to fuse it with the code graph and git timeline.

### Structure-aware code representation: the encoder backbone for the Spatial axis

This theme provides the **Baselines/Methods** for encoding the static Z axis. The bimodal and structure-aware pretrained models — CodeBERT [2002.08155], GraphCodeBERT [2009.08366] (dataflow into pretraining), UniXcoder [2203.03850] (AST linearization + cross-modal contrastive alignment), and CodeT5 [2109.00859] — are standard encoder **Baselines** for grounding code semantics, and several (CodeBERT, CodeT5) double as candidate backbones for Form B's compression. The AST/syntax-injection line — StructCoder [2206.05239] (ASTs + DFG in encoder and decoder with auxiliary structure-prediction tasks), AST-T5 [2401.03003] (AST-aware segmentation and span corruption, no architecture change), TreeBERT [2105.12485] (AST composition paths), and code2vec [1803.09473] (AST-path decomposition) — are borrowed **Methods** for the "history-aware AST" projection, with StructCoder's and AST-T5's auxiliary structure-prediction objectives being the most directly reusable precedent. The explicit-graph line — Learning to Represent Programs with Graphs [1711.00740] (unified AST + data/control-flow graph run through Gated GNNs) and PERFOGRAPH [2306.00210] (numerically-aware program graph with enriched node/edge types) — supplies the foundational graph-construction **Method/Baseline** for TAC-Graph's spatial slice and a pattern for adding TAC-specific node/edge types. Across this theme, the recurring lesson is that fusing AST with data/control-flow edges and running GNNs over the result beats flat token sequences for code understanding — the empirical foundation on which TAC-Graph adds two more axes.

### Synthesis: what TAC-Graph improves

Read together, the corpus shows that agents perceive a repository today through three *disconnected* lenses: structure-aware retrieval rebuilds a snapshot of code per query (RepoCoder, GraphCoder, DraCo, RepoGraph), agent frameworks execute and discard action loops (SWE-agent, OpenHands, Agentless), and memory systems store trajectories as task-local text (Reflexion, AWM, AriGraph). None persistently fuses *what the code is* (Z), *how it got written this session* (Y), and *how it evolved across history* (X) into one queryable object. TAC-Graph's contribution is precisely this fusion: a history-aware code graph that an agent can query via Form A or absorb via Form B, turning the per-query reconstruction these methods perform into a standing, cross-axis representation. The benchmarks here (RepoEval, RepoBench, CrossCodeEval, ReccEval, R2C2-Bench, SWE-bench, SWE-Gym, AgentBench) form a ready evaluation suite, and the structure-aware encoders and trajectory-memory methods supply the building blocks for the spatial and agentic axes respectively.

## Software Engineering Perspective: Prior Knowledge, Software Evolution & Commit Untangling

Software engineering (SE) research supplies two pillars for the TAC-Graph project: (i) a long tradition of *knowledge-grounded* code generation that retrieves external prior knowledge (docs, APIs, knowledge graphs) to condition synthesis — directly motivating the implicit prior-knowledge delivery of Form B — and (ii) a mature line of work on *mining software repositories (MSR)*, *code-change representation*, and *commit untangling*, which both populates the Temporal (X) axis and defines Task Untangling, a direct downstream task for the RP. We organize the SE corpus along these themes.

### Knowledge-grounded and prior-knowledge-driven code generation

A consistent SE finding is that grounding an LLM in *external* prior knowledge — rather than relying on parametric memory — sharply improves code generation, especially for fast-evolving or proprietary libraries. **DocPrompting** [2207.05987] retrieves relevant documentation given a natural-language intent and conditions generation on the docs, gaining up to 6.9% exact match on a new Bash (tldr) dataset and 2.85% pass@1 on CoNaLa. **ToolCoder** [2305.04032] instead teaches the model to *invoke* an API-search tool during decoding (training data auto-annotated by ChatGPT), yielding at least 6.21% average pass@1 and 9.64% pass@10 across five benchmarks. **CodeAgent** [2401.07339] wraps five repository-interaction tools (symbol navigation, test execution, etc.) into an agent and introduces **CodeAgentBench** (101 repo-level samples from five Python projects), reporting 18.1%–250% gains and surpassing Copilot on repo-level tasks. *Value to us: borrowed Method (retrieval/tool-use as the injection mechanism Form A's DSL would automate); Benchmark (CodeAgentBench).* These works establish the core thesis behind Form B — that compactly delivering the *right* prior knowledge beats brute-forcing context.

A second cluster grounds generation in **knowledge graphs** of code/APIs, the closest SE analog to TAC-Graph's structured tensor. **Context-Augmented Code Generation Using Programming Knowledge Graphs** [2410.18251] builds a Programming Knowledge Graph for semantic retrieval, adding tree-pruning and re-ranking to suppress irrelevant context and hallucination (up to +20% on HumanEval, +34% on MBPP). **Knowledge Graph Based Repository-Level Code Generation** [2505.14394] represents a codebase as a dependency graph with hybrid retrieval and evaluates on the Evolutionary Code Benchmark, explicitly targeting *evolving* codebases. **APIKG4Syn** [2512.00380] constructs API knowledge graphs to *synthesize* training pairs for low-resource frameworks (HarmonyOS), using uncertainty estimation and MCTS to filter data; a fine-tuned 7B model reaches 25.0% pass@1, beating untuned GPT-4o. **KCoEvo** [2603.07581] is the most temporally explicit: its API knowledge graph encodes *version transitions*, and a two-stage "evolution-path retrieval → path-informed generation" pipeline improves migration accuracy under evolving third-party APIs. *Value to us: borrowed Method (KG construction + path retrieval and the pruning/re-ranking strategies directly inform Form A's DSL query design and the "history-aware AST"); Benchmark (Evolutionary Code Benchmark via [2505.14394]); KCoEvo and APIKG4Syn give concrete recipes for projecting API *evolution* into structured prior knowledge.* Finally, **CodeRAG-Bench** [2406.14497] is a comprehensive retrieval-augmented code-generation benchmark spanning docs, tutorials, and repositories; it confirms that high-quality retrieved context helps but exposes weak lexical overlap and poor long-context utilization. *Value to us: Benchmark / new Metric — a ready evaluation scaffold for whether Form A surfaces the right cross-axis context, and an honest baseline on retrieval-quality vs. generator-utilization gaps.*

### Mining software repositories, defect prediction, and software evolution

The MSR line motivates and instruments the Temporal (X) axis. The vision paper **DeepSoft** [1608.00092] argues for LSTM-based modeling of software-evolution history (code/task recommendation, risk identification, patch generation), foreshadowing sequence/temporal models over commit history. *Value: Theory/inspiration.* The recent survey **The Rise of Language Models in Mining Software Repositories** [2604.00787] analyzes 85 papers, taxonomizing which artifacts (commits, diffs, bug reports) LMs consume and how MSR usage evolved. *Value: Theory/inspiration + Baseline taxonomy for situating Task Untangling and code-change representation.*

Just-in-time (JIT) defect prediction shows how commit-level signals are modeled and is a natural cousin of untangling (separating buggy from safe changes). **CC2Vec** [2003.05620] learns hierarchical, attention-based diff embeddings supervised by commit messages, validated on log-message generation, patch identification, and JIT defect prediction. **SimCom++** [2403.11079] fuses hand-crafted expert features with deep commit-semantic representations (early + late fusion) for 5.7–26.9% JIT gains, an explicit "expert-knowledge-meets-DL" template. **BiCC-BERT / JIT-BiCC** [2410.12107] pre-trains bi-modally on diffs + commit messages via a "Replaced Message Identification" objective (+10.8% F1 over eight baselines on 27k+ changes). **Graph-Based JIT defect prediction** [2110.05371] mines developer-to-file *contribution graphs* from commit history, reaching F1 up to 77.55% and MCC up to 53.16% across 14 projects, showing collaboration-network structure beats intrinsic code metrics. *Value to us: borrowed Method (commit-node feature encoders for the X-axis; the contribution-graph is a literal temporal-graph instantiation); Baseline (CC2Vec); new Metrics (F1/MCC protocols for distinguishing defect-prone commits).* Bridging defects and evolution, **Evaluating SZZ Implementations** [2308.05060] empirically audits six SZZ variants on 76,046 Linux-kernel bug-fix pairs, reports that 17.47% of fixes are untraceable "ghost commits," and proposes TC-SZZ (trace-all-commits) to recover ~17.7% of failures. *Value to us: Baseline algorithm + ground-truth dataset for linking bug-inducing to bug-fixing commits along the X-axis; its function/file-level tracing maps onto the tensor's spatial granularity, and SZZ-style ground truth is essential for any Task-Untangling evaluation.*

### Commit untangling and change decomposition (direct downstream task)

This theme is the most directly tied to the RP's Task Untangling direction. Foundational empirical work quantifies the problem: **A Fine-grained Data Set and Analysis of Tangling in Bug-Fixing Commits** [2011.06244] crowdsource-labels bug-fix commits and finds only 17–32% of changed lines actually fix the bug (with ~11% inter-labeler disagreement), proving that *unvalidated bug-fix datasets are noisy* and motivating robust commit-level representations. *Value: Benchmark / Task-definition — a fine-grained labeled corpus and the strongest motivation for the untangling task.* The classic method **Untangling Fine-Grained Code Changes** [1502.06757] (EpiceaUntangler) clusters related fine-grained changes to recover atomic commits, achieving a 91% median separation rate, and ships a dataset of properly separated changes. *Value: borrowed Method + Benchmark baseline.*

The recent wave reframes untangling for LLMs. **LLM-Based Detection of Tangled Code Changes** [2505.08263] uses few-shot/CoT prompting over diffs + commit messages to flag tangling (F1 0.883; 0.906 with ML classifiers on LLM embeddings), cleaning method-level bug datasets across 49 projects. **Detecting Multiple Semantic Concerns in Tangled Code Commits** [2601.21298] recasts the task as *multi-label* classification (multiple co-occurring concerns per commit), fine-tuning LMs up to 14B on artificially tangled real-world commits and showing commit messages boost accuracy by up to 44%. **Atomizer** [2601.01233] (ICSE 2026) is a two-agent (grouper + reviewer) framework using Intent-Oriented Chain-of-Thought to infer change purposes and iteratively refine decompositions, outperforming SOTA by 6.0%/5.5% on C#/Java and >16% on complex commits. *Value to us: borrowed Method (intent-driven, multi-agent and multi-label decomposition that the RP's cross-dimension projection should subsume); Benchmark (artificially-tangled corpora, C#/Java sets); new Metric (multi-label concern F1).* Atomizer's grouper/reviewer success-vs-failure loop also doubles as inspiration for the Trajectory Optimization direction. Collectively these define both the framing (multi-label, intent-aware) and the metrics for evaluating TAC-Graph on untangling.

### Code-change representation and edit modeling

Underpinning both defect prediction and untangling is the question of how to *embed a diff*. Several pre-trained models target this directly. **CCRep** [2302.03924] (ICSE 2023) uses a pre-trained code model plus a "query-back" mechanism to isolate and encode changed fragments, beating prior work on commit-message generation, patch-correctness, and JIT defect prediction. **CCT5** [2305.10785] pre-trains a change-oriented T5 on 1.5M+ diff/message pairs with five tasks, outperforming general code models on review and change-summarization. **CCBERT** [2309.15474] (ICSME 2023) learns change representations *without* commit messages via four self-supervised objectives over original/modified tokens and edit operations, beating CC2Vec by 7.7–14.0% at far lower compute. **CommitBERT** [2105.14242] releases a 345k-pair multilingual diff→message dataset and frames commit-message generation as NMT. **CodeReviewer** [2203.09095] (ESEC/FSE 2022) pre-trains on multilingual diffs with four tasks for change-quality estimation, review-comment generation, and code refinement. *Value to us: borrowed Method + Baseline (drop-in encoders for X-axis commit/edit nodes — CCRep's query-back to localize changed fragments and CCT5/CCBERT's change-aware objectives directly seed the "history-aware AST"); Benchmark (CommitBERT and CodeReviewer corpora reusable for untangling).*

A parallel "edit-as-translation" cluster models edits structurally. **Learning to Represent Edits** [1810.13337] (ICLR 2019) pairs a neural editor with an edit encoder that captures structural/semantic change, generalizing edit patterns. **Unsupervised Learning of General-Purpose Embeddings for Code Changes** [2106.02087] learns change embeddings purely from the changes themselves (supervised only by correct application), competitive on change-application (+5.9pp) and commit-message generation. **MODIT** [2108.06645] (ASE 2021) fuses edit location, surrounding context, and commit message (intent) for top-1 patch generation, a concrete multi-modal cross-dimension-projection baseline whose commit-message-as-intent signal parallels the RP's Agentic axis. **CoditT5** [2208.05446] (ASE 2022) introduces an explicit edit-modeling pre-training objective, improving comment-updating, bug-fixing, and code review over generation-only models. **On Learning Meaningful Code Changes via NMT** [1901.09102] (ICSE 2019) shows an NMT model trained on 236k before/after PR pairs replicates developer edits in up to 36% of cases, capturing refactoring vs. bug-fix patterns. *Value to us: borrowed Method (edit encoders and edit-pretraining objectives as X-axis building blocks; MODIT's multi-modal fusion as a cross-axis projection template); Baseline for change synthesis.* **CodeT5** [2109.00859] (identifier-aware encoder-decoder) rounds out this group as a general-purpose backbone for change-oriented and untangling tasks. *Value: Baseline backbone.*

### Synthesis for TAC-Graph

The SE corpus supplies three transferable assets. First, knowledge-grounded generation — especially the KG-based works [2410.18251, 2505.14394, 2512.00380, 2603.07581] and tool/doc retrieval [2207.05987, 2305.04032, 2401.07339] — validates Form B's premise (compress structured prior knowledge and inject it) and gives Form A a vocabulary of query/pruning/re-ranking operations; KCoEvo's API-version-transition graphs are a direct prototype for fusing the Temporal axis into prior knowledge. Second, the untangling line [2011.06244, 1502.06757, 2505.08263, 2601.21298, 2601.01233] precisely defines the Task Untangling downstream task, its noisy-data motivation, multi-label framing, and concern-level metrics. Third, the change-representation models [2302.03924, 2305.10785, 2309.15474, 2105.14242, 2203.09095, 1810.13337, 2106.02087, 2108.06645, 2208.05446, 1901.09102] are off-the-shelf encoders for the X-axis commit/edit nodes, while SZZ [2308.05060] and JIT defect-prediction graphs [2110.05371, 2003.05620, 2403.11079, 2410.12107] supply ground-truth links and temporal-graph constructions. Across these, the recurring benchmarks (CodeRAG-Bench, CodeAgentBench, Evolutionary Code Benchmark, EpiceaUntangler's and the fine-grained tangling datasets, CommitBERT/CodeReviewer corpora) and metrics (pass@k, exact match/identifier F1, multi-label concern F1, JIT F1/MCC) form a concrete evaluation suite for the RP's knowledge-grounded and untangling claims.

## HCI / Security Perspective: Code & Software Visualization

The HCI and security literature contributes comparatively little in the way of benchmarks or training signals to TAC-Graph, but it is arguably the richest source of *design intuition* for how a three-axis tensor should be rendered, serialized, or surfaced to a human or an agent. These papers answer a question the AI/NLP corpus largely ignores: once we have fused temporal, agentic, and spatial information, what visual metaphor, layout, and saliency choices actually help a human (or a model trained on human signals) comprehend it? We group them into four themes: (i) 3D code-city and software-evolution visualization, (ii) program-comprehension theory and developer mental models, (iii) eye-tracking and human-attention studies, and (iv) code-as-image / binary visualization for security. We flag upfront that almost all of these are **Theory/inspiration** for us; the few that ship datasets are noted explicitly.

### 3D code-city and software-evolution visualization

The most direct precedent for a literally three-dimensional code representation is the *code-city* metaphor, which maps a codebase onto a navigable 3D city where buildings encode classes/files and districts encode packages. M3triCity [2204.10006] extends this metaphor along two of TAC-Graph's exact axes: it animates structural evolution across versions (our X/Temporal axis) and overlays data- and file-access patterns (analogous to our Z-axis call/API layer), making it the closest existing artifact to a temporal-spatial code view. Two VISSOFT 2024 systems push the metaphor toward the workflows TAC-Graph targets: one fuses static structure with dynamic execution traces inside the city while integrating Git hosting and issue trackers for code review [2408.08141], demonstrating concretely how spatial structure and temporal history can be co-rendered; the other applies the city metaphor to *collaborative* architecture-change planning [2408.16777], suggesting downstream value for our Task Untangling and Trajectory Optimization directions where teams reason about structural change over time. The systematic literature review of modern software visualization [2003.00643] is our **Baseline** map of this design space: it taxonomizes visualizations by the software aspect addressed (structure, behavior, evolution), the engineering task, the user role, and the display medium, giving us a vocabulary to justify a novel "3D code-city + evolution overlay" form rather than reinventing it. All four are **Theory/inspiration** for serialization/rendering decisions; the SLR additionally serves as a positioning baseline. The honest caveat is that code-city tools render *human-facing* views and offer no evidence that the metaphor helps a *model* — their contribution is the layout vocabulary (height/footprint/color/district encodings, version morphing) we can borrow, not an evaluation protocol.

### Immersive visualization and its evaluation

Two VR studies probe whether immersion improves comprehension of multi-dimensional repository data, which speaks to the upper bound of how much spatial encoding can help. The registered-report study on 3D/VR visualization of repository metrics in SE education [2412.16061] proposes a controlled VR-vs-non-VR experiment; we count it as **Method/Benchmark** because its experimental design is a reusable evaluation *methodology* for testing whether any TAC-Graph rendering actually aids developer understanding, even though it provides no dataset we would consume directly. On the security side, SecCityVR [2504.18238] renders vulnerabilities as colored building floors inside a code city with integrated call graphs and supports multi-user audits; its user study found higher usability and lower frustration than dashboards but *longer* task times — a candid, useful warning (**Theory/inspiration**) that immersive 3D buys engagement and spatial recall at a cost in raw speed, which should temper any claim that 3D-ness alone is a win for TAC-Graph.

### Program comprehension: what structure humans actually use

A second cluster grounds *which* parts of the spatial axis matter to humans, informing both node saliency in the tensor and the Form A query interface. Wyrich's conceptual model of source-code comprehension [2310.11301] is **Theory/inspiration** that supplies a formal account of comprehension goals; knowing what cognitive targets drive understanding helps specify what a `Query(Target, History, Agent_Trace)` interface should be able to surface. The visuo-spatial mental-model study [2304.09301] compares tab-based versus canvas/spatial layouts and, notably, finds *no significant comprehension-accuracy effect* but measurable differences in how time is allocated to navigation versus annotation — another honest signal (it is **Benchmark/Theory** via its controlled-study design) that spatial layout changes *behavior* more than it changes *outcomes*. Code Compass [2405.06271] characterizes the friction of navigating unfamiliar codebases and prototypes context-aware documentation/visualization in the IDE; we treat it as **Method/inspiration** that motivates exactly the navigation-friction reduction Form A's DSL is meant to deliver by pulling precise structural context on demand.

### Eye-tracking and human attention as a design signal

The most actionable HCI inputs are eye-tracking studies that quantify *where developers look*, which can directly weight nodes/regions in our representation. The classic Java-method reading study [1903.03358] finds developers attend more to method bodies than signatures and give disproportionate gaze to call terms, with experts revisiting bodies as size grows — concrete evidence that **call-graph nodes and method bodies deserve higher salience** in the Z-axis. Follow-up Attention [2210.05506] compares developer gaze to neural-model attention and introduces a "follow-up attention" post-processing **Metric** that predicts the next line a developer examines (47% accuracy), characterizing human code-traversal that could ground the read/explore operations on our Y/Agentic axis. EyeTrans [2402.14096] goes further and *fuses* human gaze into Transformer attention for summarization (up to ~29.9% gain), a **borrowed Method** template for injecting perceptual-saliency weights into the cross-dimension projection that builds the history-aware AST. Two IDE-instrumentation studies characterize agent-relevant behavior: the eye-tracking + IDE-action study of validating/repairing LLM-generated code [2405.16081] documents real read/edit/search patterns (alternating between code and comments, preferring delete-and-rewrite), and the MIMESIS-based behavioral-abstraction work [2403.08480] (which we count as **Benchmark/Method/Metric** for its abstraction that compares developers across skill/style) offers a framework for distinguishing successful from unsuccessful navigation strategies — directly relevant to Trajectory Optimization. Finally, the large-scale analysis of 11,579 AI-assisted IDE sessions [2604.00436] is a genuine **Benchmark/Task-definition** contribution from this cluster: its taxonomy of behavioral shifts (progressive specification, cognitive delegation, context management) is empirical evidence for what the Y-axis must capture and motivates Form A's premise that agents will increasingly *query* history/structure rather than read raw files.

### Code-as-image and security visualization

The security sub-corpus pursues a different serialization of the spatial axis: render code or binaries as *images* and let CNNs read them. This is methodologically adjacent to Form B's goal of compressing structure into a dense, non-symbolic representation. WYSIWIM [2002.02650] converts source to images and processes them with pretrained image networks, reaching parity with AST-based encoders on clone detection — a **Baseline/inspiration** that non-symbolic visual encodings can capture code semantics, relevant to Form B's latent-embedding direction. On the malware side, several works convert binaries/bytecode to images and classify them: frequency-domain (DCT) grayscale imaging with the **MaleX Benchmark** [2101.10578], the **MalVis Benchmark** of 1.3M entropy/N-gram bytecode images [2505.12106], multi-feature-fusion RGB images stacking DEX/manifest/API channels [2408.16555], pixel-row oversampling to make vulnerability images more continuous (VulMCI) [2402.18189], and an early self-organizing-network binary-visualization detector [1904.00859]. These are mostly **borrowed Method/inspiration** for one specific idea — that heterogeneous program artifacts can be stacked as image channels and fused — which maps onto how TAC-Graph might project its three axes into a unified tensor; MaleX and MalVis additionally function as **Benchmarks**, though for malware classification rather than any TAC-Graph task. Two further papers contribute essential cautions and a graph alternative. COPYCAT [1909.09735] demonstrates executable-preserving adversarial attacks (~98.9% misclassification) on visualization-based detectors — a **Theory/inspiration** robustness warning for any image- or graph-based code representation we build. Devign [1909.03496] is the most directly reusable security paper: it is a **Method/Benchmark/Baseline** that jointly encodes AST, CFG, DFG, and token sequence into a unified GNN, which is essentially a concrete instantiation of our Z-axis spatial-graph construction and a proven baseline architecture for learning over multi-relational program graphs.

### Takeaway for TAC-Graph

Collectively this perspective supplies metaphors, layout primitives, and human-perception priors rather than benchmarks: the code-city encoding vocabulary (M3triCity [2204.10006], the VISSOFT systems [2408.08141; 2408.16777], the SLR [2003.00643]) for *how* to render the temporal-spatial slices; gaze-derived saliency (call terms and method bodies [1903.03358], follow-up attention [2210.05506], EyeTrans's gaze-into-attention fusion [2402.14096]) for *which* nodes to weight; large-scale behavioral data ([2604.00436; 2405.16081; 2403.08480]) for *what* the agentic axis and Form A query interface must capture; and the code/binary-as-image line (WYSIWIM [2002.02650], MalVis [2505.12106], Devign [1909.03496]) for *channel-stacking* serialization of heterogeneous artifacts. Equally valuable are the honest negative and cautionary results — no accuracy gain from spatial layout [2304.09301], slower task completion in VR [2504.18238], and adversarial fragility of image-based detection [1909.03496; 1909.09735] — which collectively warn that 3D-ness and visual encoding are not free wins and must be justified by measured comprehension or downstream-task gains rather than asserted.

## Methodological & Theoretical Toolbox: Temporal GNNs, Graph-VAE & Context Compression

The methods and theory underpinning TAC-Graph come almost entirely from outside the software-engineering literature: dynamic-graph learning, graph autoencoders, graph transformers, and parameter-efficient prompting. We survey each family below and pin it to the specific TAC-Graph component it enables. Every paper in this group is a borrowed Method or Theory/inspiration (several also double as a Baseline or Benchmark), since none was designed for code evolution; the contribution of TAC-Graph is to repurpose and fuse them across its three axes.

### Temporal / Dynamic GNNs: the engine for the X-axis and cross-axis time-granularity mismatch

The central modelling challenge of TAC-Graph is that its axes tick at incompatible rates: agent operations occur in seconds (Y), commits in days (X), while the spatial graph (Z) is quasi-static. Temporal GNNs supply the borrowed Method for representing a graph whose nodes, edges, and features change over continuous time. The two canonical formulations are **TGAT** [2002.07962], which introduces a temporal graph attention layer plus a *functional time encoding* derived from Bochner's theorem, and **TGN** [2006.10637], which augments graph aggregation with a per-node *memory module* summarising interaction history and unifies many earlier models as special cases. These two give TAC-Graph its core Temporal-GNN backbone: TGAT's continuous time encoding is precisely what is needed to place second-scale agent events and day-scale commits on a common temporal manifold, and TGN's memory is the natural store for per-node (per-file, per-function) evolution state. Earlier dynamic-embedding methods are Theory/inspiration and Baselines for the same component: **DyRep** [1803.04051] models structural and communication processes with a temporal point process explicitly across *different time scales*, directly motivating TAC-Graph's seconds-vs-days fusion; **JODIE** [1908.01207] couples RNNs and adds a projection operator to *forecast* future embeddings, suggesting how to predict future code states from history (relevant downstream to Trajectory Optimization); and **EvolveGCN** [1902.10191] evolves the GNN *parameters* themselves via an RNN, a useful Baseline when the node set (files, functions) changes substantially between commits.

A second cluster informs architectural and granularity choices for the Temporal-GNN. **GraphMixer** [2302.11636] (and its successor DyGFormer) shows that a simple MLP-based link encoder with neighbor mean-pooling can match complex temporal models, a sobering Baseline that argues against over-engineering the X-axis encoder. **DyGFormer** with the **DyGLib** library [2303.13047] provides both a Transformer architecture with neighbor co-occurrence encoding/patching and a reusable Benchmark scaffold for validating our temporal components. **ROLAND** [2208.07239] recurrently updates hierarchical node states and contributes a *live-update* evaluation protocol that mirrors the streaming arrival of agent-trajectory events on the Y-axis. Most directly on point, **"Exploring Time Granularity on Temporal Graphs"** [2311.12255] benchmarks DGNNs across four time granularities and finds that memory mechanisms and granularity selection are decisive — exactly the design question TAC-Graph faces when reconciling commit and agent timescales; we treat it as both Method and Benchmark guidance. Two surveys provide the Theory/inspiration that frames all of the above: the JMLR survey on dynamic-graph representation learning [1905.11485], whose encoder-decoder taxonomy also maps cleanly onto Form B, and the more recent DGNN survey [2304.05729], whose design guidelines for handling the temporal dimension inform how we fuse mismatched granularities across the three axes.

### Graph Transformers: attention over code structure and temporal topology

Graph transformers are the borrowed Method for running attention over the spatial (AST/CFG/DFG/call) graphs of the Z-axis and over the temporal commit graph of the X-axis, where sequence-style positional encodings are inappropriate. The foundational adaptation is Dwivedi & Bresson's **generalization of transformers to graphs** [2012.09699], which replaces sequential positions with Laplacian-eigenvector positional encodings and conditions attention on edge features — a Baseline architecture for any TAC-Graph attention layer. **Graphormer** [2106.05234] adds centrality, spatial, and edge structural encodings and, importantly for us, *proves that several GNN variants are special cases of attention*, supplying Theory/inspiration that unifies our spatial and temporal layers and grounds the Form B encoder. **GraphGPS** [2205.12454] offers the most practical recipe — a modular combination of local message-passing and global attention with linear complexity — and is the Method we would most likely adopt for the Z-axis encoder, with its positional-encoding taxonomy also informing Form B. **GROVER** [2007.02835] demonstrates a large self-supervised graph transformer with multi-level pretext tasks; it is Theory/inspiration for pretraining a graph encoder over code structure without labels.

### Graph Autoencoders / VAE and Graph SSL: the compression core of Form B

Form B compresses the entire 3D tensor into a latent embedding, and graph (variational) autoencoders are its direct technical foundation. The canonical **Variational Graph Auto-Encoder (VGAE)** [1611.07308] — a GCN encoder with an inner-product decoder over a continuous latent space — is the foundational borrowed Method for Form B and a Baseline for link prediction across our spatial and temporal edges. **GraphVAE** [1802.03480] extends this from embedding to *generation* of small graphs via a one-shot probabilistic decoder with graph-matching training, giving the canonical architectural Baseline for encoding heterogeneous code graphs with temporal and agentic annotations. **DiffPool** [1806.08804] contributes differentiable hierarchical pooling, the borrowed Method for coarsening AST/CFG/call graphs into compact latent codes — essential for fitting a whole repository's structure into a bounded prefix. The masked-autoencoder line provides modern, scalable encoders: **GraphMAE** [2205.10803] reframes graph SSL around *feature* reconstruction with masking, and **GraphMAE2** [2304.04779] adds re-mask and latent-prediction decoding regularizers that scale to billion-edge graphs; both are borrowed Methods and Baselines for the Form B encoder, with their reconstruction objectives directly applicable to rebuilding the spatial and temporal layers from a compressed code. The **Graph SSL survey** [2103.00111] is the Theory/inspiration that organizes generation-based, contrastive, auxiliary-property, and hybrid pretext tasks under one framework, guiding how Form B is pretrained over the tensor's axes without supervision.

### Soft-Prompt / Prefix-Tuning / Context Compression: injecting the latent into the agent

Once Form B has produced a latent embedding, it must be loaded into a frozen coding LLM without consuming the context window; parameter-efficient prompting supplies the injection mechanism. **Prefix-Tuning** [2101.00190] and **Prompt Tuning** [2104.08691] are the foundational borrowed Methods and Baselines: they show that a small set of learned continuous vectors prepended to a frozen LLM can rival full fine-tuning, which is precisely how the compressed tensor is delivered as a prefix rather than as tokens. **P-Tuning v2** [2110.07602] extends this to *deep* prompts inserted at every transformer layer, a stronger Baseline for conditioning a large coding agent on graph-VAE-derived embeddings. **Knowledge Prompts** [2210.04726] provides the Theory/inspiration that *structured* knowledge (a close analogue of git history and agent trajectories) can be distilled into soft prompts and injected into a frozen LM.

A complementary set of context-compression methods bridges the gap between a long, heterogeneous history and a short soft prefix — the practical heart of Form B's "bypass the context window" claim. **Gist Tokens** [2304.08467] compress a prompt into a few cacheable learned tokens (up to 26x), whose caching property maps neatly onto reusing the static Z-axis structure across many agent queries. **AutoCompressors** [2305.14788] accumulate long contexts into summary vectors via segment-wise training, aligning with the heterogeneous-granularity segments of the X and Y axes. The **In-context Autoencoder (ICAE)** [2307.06945] compresses context into memory slots using an autoencoding objective — a concrete blueprint for aligning Graph-VAE latents with an LLM's token space, and the closest single analogue to Form B's encode-then-inject pipeline. **xRAG** [2405.13792] compresses an entire document to *one token* through a trained modality bridge over frozen retriever and LLM; its multimodal-fusion paradigm directly informs how embeddings from the three different axes can be projected into a shared LLM space. Two token-pruning approaches round out the family as Methods and Baselines: **LLMLingua** [2310.05736], a budget-controlled coarse-to-fine compressor (up to 20x), and **SoftPromptComp** [2404.04997], which combines natural-language summarization with soft-prompt compression. All of these are borrowed Methods; collectively they establish that the full temporal-agentic-contextual history can be distilled into a compact prefix without exhausting the agent's context budget.

In sum, the methods axis of TAC-Graph is assembled from four mature toolkits: dynamic-graph learning (TGAT/TGN and the granularity literature) for the X-axis and cross-axis time fusion; graph transformers (GraphGPS/Graphormer) for attention over Z- and X-axis topology; graph (V)AEs and graph SSL (VGAE/GraphVAE/DiffPool/GraphMAE) for Form B's latent compression; and prefix-tuning plus context compression (Prefix-Tuning/ICAE/xRAG/Gist) for injecting that latent into a frozen coding agent. None targets code evolution directly, so each enters our survey as a borrowed Method or Theory/inspiration; TAC-Graph's novelty lies in welding them across three axes that no prior method jointly addresses.

## Candidate Tasks, Benchmarks, Metrics & Baselines

The TAC-Graph proposal currently specifies only a *representation* (a tensor over temporal, agentic, and spatial axes) and not a *task* on which it can be trained and measured. This section answers four practical questions for the project: (1) which downstream tasks the representation can be applied to, (2) which benchmarks support those tasks, (3) which metrics quantify progress, and (4) which prior systems constitute baselines. For each entry we flag its value to the project as one of: *borrowed Method*, *Benchmark*, *new Metric*, *Baseline*, or *Theory/inspiration*.

### Candidate Downstream Tasks

The corpus suggests five task families that exercise different combinations of TAC-Graph's three axes. Each is the most concrete way to convert the representation into a measurable contribution.

**(T1) Repository-level code completion (Z-axis, with Form A delivery).** This is the most mature and best-tooled task in the corpus, and the natural first target because it isolates the spatial axis. The task is to predict the next line, API invocation, or function body given cross-file context. It is *defined* by RepoBench [2306.03091] (three sub-tasks: retrieval, next-line prediction, end-to-end pipeline; *Benchmark*), CrossCodeEval [2310.11248] (cross-file completion where static analysis pinpoints where cross-file context is required; *Benchmark*), and RepoCoder [2303.12570] (iterative retrieve-then-generate; *borrowed Method* + *Benchmark* via RepoEval). A large body of methods provides graph- and dataflow-based instantiations of exactly the Z-axis TAC-Graph proposes: GraphCoder [2406.07003] (Code Context Graph over control/data dependence), DraCo [2405.19782] (dataflow-guided graph; also ships ReccEval), RepoHyper [2403.06095] (Repo-level Semantic Graph with search-expand-refine), CoCoMIC [2212.10007] (joint in-file/cross-file via CCFINDER), RepoFusion [2306.10998] (training on Stack-Repo context), A^3-CodGen [2312.05772] (local/global/library context), R2C2-Coder [2406.01359] (perturbation-based prompts + R2C2-Bench), Repoformer [2403.06095→2403.10059] (selective retrieval; its self-evaluation of *when* retrieval helps is directly reusable for T4), and the learned repo-prompt policy of [2206.12839]. Knowledge-graph variants — [2410.18251], [2505.14394] (Evolutionary Code Benchmark), [2512.00380], and KCoEvo [2603.07581] — extend this with API/library structure and version transitions, the latter two bridging into the X-axis. All are *borrowed Method*; several double as *Benchmark*. Documentation-grounded generation (DocPrompting [2207.05987], ToolCoder [2305.04032]) is a related API-knowledge variant.

**(T2) Issue-to-patch resolution / SWE-bench-style agentic SE (Y + Z axes, end-to-end).** Here the agent must read an issue, localize relevant code, edit across files, and pass tests — exercising the agentic and spatial axes jointly and the most direct showcase for the full tensor. Defined by SWE-bench [2310.06770] (*Benchmark*) and trained/evaluated on SWE-Gym [2412.21139] (executable runtimes + released trajectories; *Benchmark* + *Baseline*). Agent systems that produce the Y-axis trajectory data and serve as *Baselines*: SWE-agent [2405.15793] (agent-computer interface), AutoCodeRover [2404.05427] (AST-based search + spectrum fault localization), Agentless [2407.01489] (localize-repair-validate pipeline), OpenHands [2407.16741] (open agent platform), LingmaAgent [2406.01422] (repo knowledge graph + MCTS), Lingma SWE-GPT [2411.00622] (process-centric LM), CodeAgent [2401.07339] (tool-integrated, with CodeAgentBench), and CodeAct [2402.01030] (executable-code action space + CodeActInstruct dataset). RepoGraph [2410.14684] is a plug-in code graph that boosts several of these and is the closest existing analog to TAC-Graph's spatial layer used by an agent.

**(T3) Commit/PR untangling and concern detection (X-axis; the project's named "Task Untangling").** The task is to split a tangled commit (bug-fix + feature + refactor) into atomic concerns. Defined and motivated by [1502.06757] (EpiceaUntangler + dataset; *Benchmark* + *borrowed Method*) and the tangling-prevalence study [2011.06244] (*Benchmark*, quantifying that only 17–32% of changed lines in bug-fix commits address the bug). Recent LLM-based methods are strong *Baselines*: Atomizer [2601.01233] (grouper/reviewer agents with intent-oriented CoT), [2505.08263] (LLM tangled-change detection, F1 0.883–0.906), and [2601.21298] (reframed as multi-label concern detection). This task draws directly on code-change representations (see T5) and on the cross-dimension projection idea.

**(T4) Agent trajectory optimization / search pruning (Y-axis; the project's named "Trajectory Optimization").** The task is to prune wasteful agent search by contrasting successful vs. failed trajectories. There is no single benchmark, but the methods that *define* the mechanism are *borrowed Method* / *Theory/inspiration*: Reflexion [2303.11366] (verbal RL over success/failure), LATS [2310.04406] (MCTS over agent decisions), Voyager [2305.16291] (skill library), ExpeL [2308.10144] and Agent Workflow Memory [2409.07429] (induce reusable workflows from past traces), Synapse [2306.07863] (trajectory-as-exemplar retrieval), Generative Agents [2304.03442] and AriGraph [2407.04363] (episodic+semantic memory graph), and FireAct [2310.05915] (fine-tuning on trajectories). Repoformer's selective-retrieval self-evaluation [2403.10059] is reusable here as a "when to act" signal. Evaluation can borrow AgentBench's [2308.03688] failure-mode taxonomy (*Benchmark*).

**(T5) Defect localization / just-in-time defect prediction (X + Z axes).** A supervised, lower-risk task that validates the temporal axis. Vulnerability detection on code graphs is anchored by Devign [1909.03496] (joint AST/CFG/DFG GNN; *Benchmark* + *Baseline*) and the program-graph foundation [1711.00740]. JIT defect prediction (predicting whether a commit is defect-prone) is supported by [2110.05371] (contribution graphs), SimCom++ [2403.11079], BiCC-BERT [2410.12107], and CC2Vec [2003.05620]. The SZZ evaluation [2308.05060] supplies the standard algorithm and ground-truth procedure for linking bug-fixing to bug-inducing commits (*Baseline* + *new Metric*). DeepSoft [1608.00092] is *Theory/inspiration* for predictive temporal analytics.

**Recommendation for the project.** The lowest-risk, fully-tooled first task is T1 on RepoEval/CrossCodeEval/RepoBench, where graph baselines (GraphCoder, DraCo, RepoHyper) already isolate the Z-axis and the only novelty TAC-Graph must demonstrate is that *adding* the X (temporal) and Y (agentic) axes improves EM/identifier-F1 over them. The highest-impact task is T2 (SWE-bench resolved-rate), where TAC-Graph's Form A query interface competes against RepoGraph and agent baselines; T3 and T4 are the two tasks the proposal explicitly names and where the X- and Y-axis contributions are most distinctively the point of the representation.
